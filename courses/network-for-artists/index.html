아티스트를 위한 네트워크

설치미술, 라이브 퍼포먼스, 인터랙티브 작업에서 디바이스와 소프트웨어가 대화하는 방법을 함께 알아보자. 이론보다 감각을 먼저 — 직접 만져보며 이해하는 네트워크.

이 특강에서 다루는 것
  • ● IP, Port, TCP/UDP — 네트워크의 기본 언어를 아티스트 눈높이로 이해한다
  • ● OSC — 미디어아트에서 가장 많이 쓰이는 실시간 통신 프로토콜을 직접 만져본다
  • ● MQTT — IoT 설치작업의 핵심 메시징 시스템을 실제 브로커와 연결해본다
  • ● WebSocket, Serial, NDI, DMX — 상황에 맞는 프로토콜을 선택할 수 있게 된다
  • ● 네트워크 토폴로지 — 나만의 설치 네트워크를 설계할 수 있다

00 왜 아티스트에게 네트워크가 필요한가

하나의 컴퓨터에서 모든 것을 처리하던 시절은 지났다. 지금의 미디어아트는 여러 장치, 여러 소프트웨어, 여러 사람이 동시에 대화해야 한다. 센서가 읽은 값을 영상에 반영하고, 조명을 제어하고, 소리를 바꾸는 — 이 모든 흐름이 네트워크 위에서 일어난다.

생각해보기: TouchDesigner에서 만든 영상이 4대의 프로젝터로 동시에 나가야 한다면? 관객의 움직임을 카메라로 인식해서 조명과 소리를 바꾸려면? 이 모든 것의 기초가 네트워크다.

01 IP/Port — 주소와 문 번호

네트워크에서 데이터를 보내려면 어디로(IP) 보내고, 어떤 문(Port)으로 들어갈지 알아야 한다. 아파트 주소(IP)와 호수(Port)라고 생각하면 된다.

IP 주소의 영역

네트워크에는 세 가지 범위가 있다. 가장 안쪽부터 보자:

LAN (같은 공유기)

localhost (127.0.0.1) — 내 컴퓨터 안에서 프로그램끼리 통신할 때 쓰는 주소. TouchDesigner와 Max/MSP가 같은 컴퓨터에서 돌아간다면, 서로 localhost로 데이터를 주고받을 수 있다.
LAN (192.168.x.x) — 같은 공유기(와이파이)에 연결된 기기들의 주소. 전시장에서 여러 컴퓨터를 연결할 때 이 범위의 주소를 쓴다. 인터넷 없이도 작동한다!

02 TCP vs UDP — 택배와 라이브 방송

데이터를 보내는 방식에는 크게 두 가지가 있다. TCP는 택배처럼 확실하게 전달하고, UDP는 라이브 방송처럼 빠르게 흘려보낸다.

TCP (택배) UDP (라이브방송)
연결연결 필요 (악수)바로 전송
순서보낸 순서대로 도착순서 보장 안 됨
손실재전송으로 복구잃어버리면 끝
속도상대적으로 느림매우 빠름
쓰임새웹, 파일 전송, 채팅OSC, 영상 스트리밍, 게임

패킷 레이스 시뮬레이터

TCP와 UDP 패킷이 동시에 출발한다. 패킷 손실률을 조절해보면서 차이를 느껴보자.

20%
아티스트에게 중요한 포인트: 라이브 퍼포먼스에서는 "늦게 도착하는 데이터"가 "안 온 데이터"보다 나쁘다. 그래서 OSC는 UDP를 쓴다 — 놓친 프레임은 버리고 다음 프레임으로 넘어가는 게 낫다.
03 OSC — 아티스트의 만능 리모컨

OSC(Open Sound Control)는 UC 버클리에서 만들어진 프로토콜로, MIDI의 한계를 넘기 위해 탄생했다. 이름에 "Sound"가 있지만, 실제로는 모든 종류의 실시간 데이터를 주고받는 데 쓰인다. TouchDesigner, Max/MSP, Processing, Resolume — 미디어아트 소프트웨어 대부분이 OSC를 지원한다.

OSC 메시지 구조

OSC 메시지는 주소(Address)와 값(Arguments)으로 구성된다. 웹 URL처럼 슬래시(/)로 계층을 구분한다.

MIDI vs OSC: MIDI는 값이 0~127까지만 가능하고 채널도 16개뿐이다. OSC는 실수(float)를 쓸 수 있고, 주소를 원하는 대로 만들 수 있어서 훨씬 유연하다. 게다가 네트워크를 타기 때문에 케이블 없이도 된다.
04 MQTT — 센서 네트워크의 우체국

MQTT는 IoT(사물인터넷)의 표준 프로토콜이다. OSC가 "직접 전화"라면, MQTT는 "우체국(Broker)"을 통한 편지 시스템이다. 보내는 사람(Publisher)과 받는 사람(Subscriber)이 서로를 몰라도, 우체국만 알면 된다.

Pub/Sub 구조

센서 (Publisher)  →  [MQTT Broker]  →  영상 앱 (Subscriber)
Arduino  →  [Mosquitto]  →  TouchDesigner

라이브 MQTT 데모

토픽(Topic) 구조

MQTT의 토픽은 OSC 주소처럼 슬래시(/)로 계층을 나눈다. 와일드카드를 쓰면 여러 토픽을 한번에 구독할 수 있다.

토픽 의미
art/zone1/motion Zone 1의 움직임 데이터
art/zone1/+ + = zone1 아래의 모든 하위 토픽 1개 레벨
art/# # = art/ 아래의 모든 토픽 (다중 레벨)
OSC vs MQTT: OSC는 1:1 직접 전송(속도 우선), MQTT는 중앙 브로커를 통한 다:다 통신(안정성 우선). 센서가 많고 여러 앱이 같은 데이터를 써야 할 때는 MQTT, 라이브 퍼포먼스에서 빠른 반응이 필요할 때는 OSC.
05 WebSocket — 브라우저의 실시간 채널

일반적인 웹(HTTP)은 "질문 → 대답"의 반복이다. 매번 새로 연결하고, 서버가 먼저 말을 걸 수 없다. WebSocket은 한 번 연결하면 양쪽 모두 자유롭게 데이터를 보낼 수 있는 상시 채널이다.

HTTP vs WebSocket 비교

버튼을 눌러 두 방식의 차이를 시각적으로 비교해보자.

HTTP (요청-응답)
WebSocket (양방향)
브라우저에서 WebSocket을 쓰는 이유: 브라우저는 보안상 TCP/UDP 소켓을 직접 열 수 없다. WebSocket은 브라우저가 유일하게 쓸 수 있는 실시간 양방향 프로토콜이다. 그래서 p5.js에서 OSC를 쓰려면 WebSocket 브릿지가 필요하고, MQTT도 브라우저에서는 WebSocket 위에서 동작한다.

06 Serial — 아두이노와 대화하기

네트워크 프로토콜은 아니지만, 물리 세계의 데이터를 디지털로 가져오는 첫 번째 관문이 바로 시리얼 통신이다. 아두이노에 연결된 센서 값이 USB 케이블을 타고 컴퓨터로 들어오는 길.

센서 → Arduino → USB (Serial) → Processing / p5.js / TD → 시각화 / 사운드

Serial을 네트워크로 확장하기

아두이노 데이터를 네트워크로 보내는 방법은 여러 가지다:

  • Serial → OSC 브릿지: Node.js 스크립트가 Serial 데이터를 읽어서 OSC로 변환
  • WiFi 직접 전송: ESP32/ESP8266이 WiFi를 통해 직접 OSC/MQTT 전송
  • Processing 중계: Processing이 Serial을 읽고, 가공해서 OSC로 전달
07 NDI / Syphon / Spout — 영상 공유

텍스트 데이터 몇 바이트를 보내는 것과 초당 60프레임의 풀HD 영상을 보내는 것은 완전히 다른 문제다. 영상 공유에 특화된 프로토콜들이 있다.

프로토콜 범위 플랫폼 레이턴시 원리

Spout 같은 컴퓨터 Windows 거의 0 GPU 메모리 직접 공유
NDI 네트워크 Mac + Win 100~200ms 압축 → 이더넷 전송

08 ArtNet/DMX — 빛을 제어하는 언어

무대 조명, LED 스트립, 무빙 라이트 — 이 모든 것을 제어하는 표준이 DMX512이다. 하나의 DMX 유니버스는 512개의 채널을 가지고, 각 채널은 0~255 값을 보낼 수 있다. ArtNet은 이 DMX 데이터를 네트워크(이더넷)로 보내는 방법이다.

512채널 DMX 시뮬레이터

셀을 클릭하고 슬라이더로 값을 조절해보자. 처음 3채널은 RGB 조명 프리뷰에 연결되어 있다.

DMX Universe 1 선택: Ch 1
Ch 1 (R): 0
Ch 2 (G): 0
Ch 3 (B): 0
0
DMX 주소 계산: RGB 조명 1개는 보통 3채널(R, G, B)을 쓴다. 시작 주소가 1이면 Ch1=Red, Ch2=Green, Ch3=Blue. 두 번째 조명은 Ch4부터 시작. 512채널이니 RGB 조명은 최대 170개까지 제어할 수 있다.
09 네트워크 토폴로지 — 설치 설계

지금까지 배운 프로토콜들을 조합해서 실제 설치 작업의 네트워크를 설계해보자. 기기들을 어떻게 연결하느냐에 따라 안정성, 속도, 유지보수 난이도가 달라진다.

토폴로지 빌더

노드를 추가하고 드래그해서 배치해보자. 프리셋으로 일반적인 설치 구성을 확인할 수 있다.

Star vs Mesh: Star(별 모양)는 중앙 컨트롤러가 모든 것을 관리하므로 간단하지만, 중앙이 죽으면 전체가 멈춘다. Mesh(그물 모양)는 기기들이 서로 직접 통신하므로 하나가 죽어도 나머지가 살아있지만, 설정이 복잡하다.

설치 시나리오 예시

시나리오 구성 프로토콜 토폴로지
인터랙티브 전시 카메라 + PC + 프로젝터 2대 OSC + Syphon/Spout Star
IoT 환경 설치 센서 10개 + 서버 + LED 5줄 MQTT + ArtNet Star (브로커 중심)
라이브 퍼포먼스 음악 + 영상 + 조명 OSC + NDI + DMX Star (메인 PC 중심)
관객 참여 웹앱 스마트폰 N대 + 서버 + 메인 스크린 WebSocket Star (서버 중심)
